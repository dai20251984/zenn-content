## はじめに

介護施設の共用フロアは音声認識にとって過酷な環境だ。テレビ、スタッフ同士の会話、入居者の呼びかけという3種のノイズが常時混在している。

本記事では、Androidで動作するハンズフリー音声AIインカムの開発過程で直面したノイズ問題と、RMSエネルギーゲート（L2）とSilero VAD（L3）を中心とした5層防御の実装詳細を解説する。閾値がどのように決まったかという「失敗の軌跡」も含めて記述する。

背景・ストーリーはnoteに → https://note.com/yamashita_aidev/n/nb325c29c62b3

---

## 解決すべき問題

### 介護現場の3種混合ノイズ

音声認識における「環境ノイズ」は単一種類ではない。介護施設フロアには以下の3種が同時に存在する。

| ノイズ種別 | 特徴 | 難しさ |
|-----------|------|--------|
| テレビ音声 | 会話と同リズム、終日継続 | 人の声と周波数帯が重なる |
| スタッフ間会話 | 音量・距離が近い | 目的の発話と区別困難 |
| 入居者の呼びかけ | 意味のある言葉だが応答不要 | コンテキストのみで判別可能 |

会議室で測定した認識率95%超という値は、この環境に持ち込んだ瞬間に崩れた。

### ウェイクワードだけでは足りない理由

「ウェイクワード検出後の質問待ち状態」で、テレビ音声がそのまま質問として送信される問題が頻発した。ウェイクワード誤検知よりも、ウェイクワード後の状態でのノイズ混入の方が発生頻度が高かった。

計測した結果、サーバーへの全送信のうち40%以上がノイズ起因だった。これは直接的なAPIコスト増加を意味する。

## 5層防御アーキテクチャ

1層のフィルタリングでは対処できないと判断し、5層の独立したフィルタを設計した。各層が前の層の見逃しを補完する。

| レイヤー | 担当 | 手法 | 主な対象ノイズ |
|--------|------|------|-------------|
| L1 | Android OS | VOICE_RECOGNITION + NoiseSuppressor | 環境ノイズ全般 |
| L2 | Android App | RMSエネルギーゲート（閾値: 5） | 環境ノイズ・静寂時残留ノイズ |
| L3 | VAD | minSpeechFrames=3、threshold=0.60 | 瞬間ノイズスパイク |
| L4 | Server Python | SNRチェック（RMS < 200） | L2-L3を通過した低エネルギー音声 |
| L5 | Android App | 未知語フィルタ + 1文字完全一致 + 後方一致 | 音声認識エンジンのコマンド誤検知 |

重要な設計方針: **各層は独立して動作すること**。特定のレイヤーに依存する設計は、そのレイヤーが想定外の入力を受けたとき全体が崩れる。

## L2: RMSエネルギーゲートの実装と閾値導出

### なぜRMSを選んだか

L2の手法候補を比較した結果:

| 手法 | 計算コスト | リアルタイム性 | 精度 | 採否 |
|------|----------|------------|------|------|
| RMSエネルギー | 極小（加算＋平方根のみ） | 1フレーム以内 | 中 | 採用 |
| FFTスペクトル分析 | 大 | フレーム遅延あり | 高 | 不採用 |
| 機械学習ノイズ分類 | 最大 | 推論遅延あり | 最高 | 不採用 |

ハンズフリーインカムはリアルタイム性が最優先。精度の不足はL3〜L5で補完する設計にした。

### 実装コード

```kotlin
// AudioRecord のコールバック内で各フレームを評価する
// buffer: ShortArray（1フレーム分の音声データ）
private fun processAudioFrame(buffer: ShortArray) {
    // RMS（Root Mean Square）エネルギーを計算
    // 計算コスト: O(n) の加算 + 平方根 1回のみ
    val rms = kotlin.math.sqrt(
        buffer.fold(0.0) { acc, sample ->
            acc + sample.toDouble() * sample.toDouble()
        } / buffer.size
    )

    if (rms < ENERGY_GATE_RMS_THRESHOLD) {
        // 閾値未満は音声処理パイプラインに流さない
        // デバッグ用: ゲートされたフレーム数を記録
        gatedFrameCount++
        return
    }

    // 閾値以上のフレームのみ VAD および後段レイヤーへ転送
    audioRouter.routeFrame(buffer, frameCount)
    frameCount++
}

companion object {
    // 実機実測に基づく閾値（後述の失敗談を参照）
    // デバイス本体マイク使用時:
    //   - 環境ノイズ（テレビ・空調）: RMS 3〜7 程度
    //   - 静寂時の電気的残留ノイズ: RMS 1〜3
    //   - 通常発話: RMS 50〜200
    //   - ささやき声: RMS 15〜40
    // → 5.0 は「ノイズを遮断しつつささやき声を通す」境界値
    private const val ENERGY_GATE_RMS_THRESHOLD = 5.0
}
```

### 閾値が 300 → 20 → 5 と変わった理由

**RMS 300（最初の設定）**

根拠: 会議室でイヤホンマイクを使用して発話したとき、RMSは800〜3000だった。環境ノイズは高くても100程度と見積もり、300に設定した。

結果: 実機テスト初日、ほぼ全フレームがゲートされて音声がサーバーに届かなかった。

原因: 本システムはイヤホンマイクなしでデバイス本体マイクを使用していた。口元との距離が離れるため収音レベルが大幅に低下する。実測すると通常発話のRMSは50〜200しかなかった。

**RMS 20（2回目の設定）**

結果: 誤検知は減少したが、テレビ音声（現場実測でRMS 8〜15程度）が一部通過し続けた。

**RMS 5（最終値）**

介護施設フロアでの実測値:

```
電気的残留ノイズ:  RMS 1〜3
環境ノイズ（大半): RMS 3〜7
テレビ音声:        RMS 8〜15（音量により変動）
ささやき声:        RMS 15〜40
通常発話:          RMS 50〜200
```

RMS 5は「環境ノイズの大半を遮断しながら、ささやき声を通すギリギリのライン」だ。

## L3: Silero VAD パラメータのチューニング

### チューニング前後の比較

```kotlin
// === チューニング前（机上設計値）===
// 一般的な会議システム向けの推奨値に近い設定
val vad = SileroVad(
    threshold = 0.65f,       // 感度: 標準 → 現場では小声を弾いた
    minSpeechFrames = 5      // 発話開始判定フレーム数 → 応答に160msの遅延
)

// === チューニング後（実機最終値）===
// 介護現場の発話特性に合わせた調整
val vad = SileroVad(
    threshold = 0.60f,       // 感度を上げて小声を検出可能に
    minSpeechFrames = 3      // 判定を96msに短縮（64ms の体感改善）
)
```

### threshold: 0.65 → 0.60 に下げた理由

介護スタッフが小声で質問する理由は周囲への配慮だ。「褥瘡」「排泄」「経管栄養」といった医療用語を大声で発話することは現場の状況として難しい場面が多い。threshold=0.65はこの実運用を想定していない値だった。

### minSpeechFrames: 5 → 3 に変更した理由

```
1フレーム = 32ms
minSpeechFrames = 5: 発話開始判定まで 5 × 32 = 160ms 待機
minSpeechFrames = 3: 発話開始判定まで 3 × 32 = 96ms 待機
差分: 64ms の短縮
```

「まず」「次に」「えーと」といった短い前置きワードが5フレーム判定に到達する前にL3で弾かれ、後続の本題発話を別のutteranceとして分断していた。3フレームに変更してこの分断が解消された。

## L5: 音声認識エンジンのコマンド誤検知フィルタ

### 誤検知が発生した具体的なケース

質問フォローアップ受付状態で「食事介助の方法を教えて」と発話したとき、音声認識エンジンが `"[unk] [unk] 次"` と認識した。この結果から「次」が抽出され、コマンドとして誤処理された。

### 実装コード

```kotlin
// コマンド判定関数 — 音声認識結果からコマンドを識別する
// 誤検知の発生パターンを踏まえて3段階のガードを設ける
internal fun isNextCommandMatch(recognizedText: String): Boolean {

    // ガード 1: 未知語トークンを含む結果はコマンドとして扱わない
    // 理由: 長い発話が部分認識されて末尾にコマンド文字列が残るケースを排除する
    // 例: 「食事介助の方法を教えて」→ "[unk] [unk] 次" → 誤検知を防止
    if (recognizedText.contains("[unk]")) return false

    // スペース（全角・半角）を正規化してから照合する
    val normalized = recognizedText
        .replace(" ", "")
        .replace("　", "")
        .trim()

    return config.nextCommandVariants.any { variant ->
        val normalizedVariant = variant
            .replace(" ", "")
            .replace("　", "")
            .trim()

        if (normalizedVariant.length <= 1) {
            // ガード 2: 1文字コマンド（例: 「次」）は完全一致のみ許可
            // 理由: 「移乗の次」「ご飯の次に」等で誤検知が発生するため
            normalized == normalizedVariant
        } else {
            // ガード 3: 2文字以上は後方一致も許可
            // 理由: TTS音声がマイクに回り込んだとき、
            //       コマンドの前に余分な音が付く場合がある
            normalized == normalizedVariant || normalized.endsWith(normalizedVariant)
        }
    }
}
```

## 5層防御の効果測定

| 構成 | ノイズ起因の誤送信率 | 正常な質問送信率 |
|------|------------------|--------------|
| L2のみ（RMS 300）| 送信ゼロ（発話も届かない） | - |
| L2のみ（RMS 5） | 約30% | 約70% |
| L2 + L3 | 約10% | 約90% |
| L1〜L5（最終構成）| 3%未満 | 97%以上 |

40%超から3%未満への削減は、APIコスト削減として直接効いている。

---

## まとめ

1. **閾値は実機・実環境で決まる**: RMS 300→20→5という変遷が証明している。机上設計値は仮説に過ぎない。
2. **5層の独立したフィルタが必要**: 各層が前の層の見逃しを補完する。
3. **マイク構成が閾値を決定する**: イヤホンマイクとデバイス本体マイクでは収音レベルが大きく異なる。
